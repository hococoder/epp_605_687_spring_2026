Overview

iOS is a worldwide phenomenon! It is available in over 150 countries, and Asia - especially China - and Europe are the quickest growing populations for iOS adoption. Many of these potential users are NOT English speakers so to make your app available to the widest audience possible, it’s vital to get your app ready to handle different languages and locales. This is where internationalization and localization come into play.

What's the Difference?

So what’s the difference between internationalization and localization?

  • Internationalization, which is usually abbreviated "i18n", is the process of designing your app for international compatibility. This covers features such as:
    • Handling text input and output in the user’s native language.
    • Handling different time, date and number formats.
    • Using the appropriate calendar and time zones.
    • Using system API calls and best practices, such as auto layout and right to left text, that enable your app for any language.
  • Localization, which is often abbreviated "l10n", is the process of translating the app’s UI and resources into another language. This is done by:
    • Using tools such as built in API calls.
    • Using the GenStrings application to create string replacements for text in your code.
    • Using 3rd party translators to put your app’s text in other languages.

UIKit, AutoLayout and Internationalization

We didn't really get into UIKit and AutoLayout this semester, but if you do, there are some things to keep in mind - we'll talk later about how these differ in SwiftUI, if at all. Many of these best practices are built into SwiftUI's layout system, so you don't have to worry about them.

To prepare your app to use localization resources properly, you should use Auto Layout during the internationalization process. The biggest thing to key in on here is that Auto Layout allows your app’s components to reconfigure their size and location if configured properly. Some tips are:

  • Don’t constrain elements to have fixed widths - this will avoid cropping if text in another language is longer than the text is in English.
  • Make heavy use of intrinsic content sizes in your text fields and labels, which allow them to auto resize.
  • In order to handle left to right vs right to left presentations, using leading and trailing attributes in your constraints so that the auto layout system can lay them out appropriately.
  • For the internal views that are away from the boundaries, pin them to adjacent views, so as the other views get resized, the pinned views will as well.
  • You should also avoid setting the min or max sizes of windows - let things adjust as needed.

Right to Left Languages

Languages such as Hebrew and Arabic are written from right to left, unlike Latin based languages which go from left to right. iOS has built in support of bidirectional views in standard views and controls. For custom controls, however, you need to handle this feature yourself. The link here is a section from a WWDC 2013 session about making your app world ready that is very interesting.

Starting with iOS 9:

  • Standard UIKit controls automatically flip in a right to left context.
  • UIImage has a method called imageFlippedForRightToLeftLayoutDirection, which helps to flip an image when appropriate
  • UIView has some built in attributes that help specify how certain views should look when flipped.

Localization

Now let’s turn our attention to localization. Each app by default, has a base localization, which is usually in the developer’s native language. You can then add languages in the project settings, some of which are shown below. Once you do this, an .lproj directory gets added to the directory structure which contains a strings file for each storyboard, xib, or localized file in your base language directory.

If you want to manage your localizations by hand there is an Xcode tool to help called GenStrings. Once you’ve replaced all of your user facing strings with NSLocalizedString macros (in UIKit) or LocalizedStringKey objects in SwiftUI, you can run the code below to generate a strings file.

For UIKit:

find . -name \*.m | xargs genstrings -o .

For SwiftUI:

find ./ -name "*.swift" -print0 | xargs -0 genstrings -a -SwiftUI -o en.lproj

This makes a Localizable.strings file, and for each instance of NSLocalizedString or LocalizedStringKey, a comment and key-value pair like the one seen below will appear in the file. You can then add this strings file to all of your language folders, and go in and change the right hand side (the value of the key value pair) to the language appropriate string.

"key" = "value"

For larger apps, or apps where you are not an expert in the language you want to localize into, a third party service can help. Xcode helps with this by including Export for Localization and Import Localization menu items in the Editor menu (these appear when the project file is selected).

The Export menu item generates a set of files that you can send off to your translators, at which point they will return one set of files for each language you want to translate. The import menu does the opposite, and turns the translator’s files into the appropriate directories and strings files that Xcode needs to present the app in multiple languages.
