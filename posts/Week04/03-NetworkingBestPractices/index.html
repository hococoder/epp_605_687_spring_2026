Week 4 - Networking Best Practices

Overview

Let’s talk about some networking best practices.

  • Using the antenna costs both power and bandwidth, so be a courteous developer and provide a good user experience - only talk to the web when you have to in order to minimize resource usage.
  • One way to do this is batching up your requests. If you have many small requests, the antenna gets powered up and down over and over, across an extended period of time. If you batched all of your requests together, and execute them as a whole, the antenna is brought up once, the requests are executed, and then the antenna is brought down, all in a shorter amount of time. This lowers the chance that your network operation may run into interference (e.g. entering an area of low signal).
  • What you download also has an impact here - make the resources you download from the network as small as you can so the operations can finish quickly and the resource use on the device side is minimized.
  • Use a cache - grab data from the web and store it locally, so you don’t have to go back to the web and get it again. A Core Data or SwiftData database can be useful here.
  • Be prepared to handle variable network capability, something you WILL encounter. Do something like pop up an alert saying network resources are limited, and for background calls to the network, don’t retry too often - this can drain the battery.
  • You can also choose to restrict certain activities, like large downloads, to one type of network (for instance, wifi instead of cellular).
  • And the biggest best practice of all - test your code! You can use the Network Link conditioner in the simulator (you can get this along with other auxiliary tools for Xcode 26 from Apple’s developer website, and in a device you can use the built in link conditioner under Settings -> Developer -> Network Link Conditioner.
