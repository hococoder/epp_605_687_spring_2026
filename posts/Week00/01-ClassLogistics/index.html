Grading Information

Grade Composition

Your grade in this course will come from 13 Mid-week quizzes (10%), 12 programming assignments (45%), an App Development Log (15%), and a Final Project (30%). Please see the Course website for more information on each of these areas.

  • You will be able to retake quizzes! The quizzes are intended to help reinforce what you are learning that week, so I want that reinforcement to stick!
  • Your grade on the programming assignments can either be an A, B, C, or F, with possible +/-.
  • Comments and feedback will be provided, but no specific deductions will be spelled out. Specific things to complete will be included in the writeup; if you don't complete those requirements, you'll lose points! Also, don't go wild doing things NOT specified in the assignment writeup - that is a waste of both of our time!
  • Each day late will result in a 1 FULL letter grade deduction.
  • If you fail to use the Coding Conventions for the class, you will start out with an A- maximum grade.
  • Final class grades can be A, B, C, F, or I, with NO +/-, and no curve.
  • Do NOT expect an A if you start your assignments a few nights before they are due. I have graded enough last minute assignments to know when you sat down and started working on it. Start early!
  • If you are near a grade boundary, your participation in class discussion (slack, forums, even office hours!) MAY help push you over.

General Grading Guidelines

Here are some general grading guidelines.

  • If your project works, includes all required components, is on time, and has good design, you can expect an A.
  • Delays in submission, lack of effort, and program execution issues can bring your grade down to a B or C.
  • An F will result in cases of plagiarism, being 3 or more days late, showing no apparent effort, or not turning an assignment in.

A Note on Programming Assignments

When completing programming assignments (this includes the App Development log), all work MUST be your own. You may discuss concepts we have covered, but you MAY NOT discuss implementations. You must also follow coding conventions.

Other Grading Guidelines

Design, function, style and timeliness all play into your final grade.

  • Design
    • Design uses patterns discussed in class properly.
    • Design is abstracted effectively.
    • Design fits the problem.
    • Design for maintenance.
  • Function
    • Does it work properly?
    • Is all required function present?
  • Style and Coding Convention
    • Is the code readable?
    • Can we figure out what's going on by just reading the code?
    • Do you follow coding conventions?
  • Timeliness
    • Was it submitted on time?
    • 1 day late == 1 letter grade deduction.
    • 2 days late == 2 letter grade deduction.
    • 3 days late == Grade of F.

Use of Open Source

  • Small sections of code may be copied and used if proper attribution is used, and it is under the proper license.
  • For code examples from class, attribute it, specifying the module you found it in.
  • Please note that unless stated otherwise, code examples from this class are for this class ONLY and cannot be used for other purposes.
  • You may use snippets from Stack Overflow (with attribution).
  • You may not use snippets from the Apple Developer forums unless the author explicitly states the code is free to use elsewhere.
  • If you find other sources and you aren’t sure if you can use them, please ask.

Use of AI Tools

The use of any AI tools, including those built into Xcode (which are disabled by default), is explicitly prohibited in this class. You are learning the basics of iOS, and using AI will not help you with that. AI agents such as ChatGPT still spew out a lot of garbage and VERY wrong code, and you don't have the basics in hand to tell the difference between reality and hallucinations. If I find that you have used AI in an assignment, I will fail you for that assignment.

### Coding Conventions Now, let’s spend a few minutes discussing Coding Conventions. Why should you use them?

  • Because we read your code, and that takes time!
  • Sometimes your code doesn’t work correctly and we will try to find out why and give partial credit when we can.
  • We also have to do this for many students, across many assignments during the semester.

Here are some of the conventions I’d like you to use.

  • Be consistent in your spacing – 4 spaces, or tabs, but not both.
  • Class names should be upper camel case, such as “HistoryViewController”.
  • Method names should be descriptive – we’ll talk later about how to build method names in Swift, but if you do it right it will be very natural and readable.
  • Variable names should be lower camel case.
  • Be descriptive – full words should be used where you can, and you should avoid abbreviations (loop indices or standard acronyms are OK).
  • In your Xcode project, your organization identifier should read “edu.jhu.ep.lastname.firstname”, and when you name the project, please call it “Lastname_Homework#”, where # is the number of the homework assignment. This helps me keep them in order when I grab them from Canvas to grade.
    • See homework 1 for a video on how to update your Xcode project to reflect your name. Please do this for all assignments.
